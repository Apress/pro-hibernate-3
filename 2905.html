Chapter 9

Deadlocks pp164 - 170

There are two serious bugs in the example. Please note that these are not caused by Hibernate itself, which is behaving entirely as specified.

The symptoms are that the example will not necessarily deadlock, and may leave the database in an inconsistent state.

Firstly we did not correctly set the level of isolation in the example. We have amended the run method of the GenerateDeadlock class as follows to correct this problem:

 
      public void run() {
 
         Session session = sessions.openSession();
 
         Transaction tx = null;
 
         try {
 
            // Set serialization level.
 
            session.connection().setTransactionIsolation(Connection.TRANSACTION_SERIALIZABLE);
 
 
 
            System.out.println(taskName + " begins a transaction.");
 
            tx = session.beginTransaction();
 
 
 
            // Step 1
 
            System.out.println(taskName + " step 1");
 
            step1(session);
 
            System.out.println(taskName + " step 1 done");
 
 
 
            // Pause to ensure proper ordering
 
            // of the steps. You would never
 
            // normally do this - however, for our
 
            // purposes it prevents either transaction
 
            // from completing before the other has
 
            // got under way (which would prevent the
 
            // deadlock from arising).
 
            pause();
 
 
 
            // Step 2
 
            System.out.println(taskName + " step 2");
 
            step2(session);
 
            System.out.println(taskName + " step 2 done");
 
 
 
            System.out.println(taskName + " committing");
 
            tx.commit();
 
            System.out.println(taskName + " committed");
 
 
 
            tx = null;
 
         } catch (Exception e) {
 
                             e.printStackTrace();
 
            if (tx != null)
 
               tx.rollback();
 
            System.out.println(taskName
 
                  + " rolled back its transaction: " + e);
 
         } finally {
 
            System.out.println("Session for " + taskName + " closed.");
 
            session.close();
 
         }
 
      }
 

Note that in the above we have also removed a superfluous call to the flush() method.

Secondly we failed to take into account one of the idiosyncracies of the Ant build tool. By default the Ant <java ... /> task kicks off the code in-thread. While this has performance advantages on large builds, it does mean that Ant only waits for the termination of the thread it started, not that of any daughter threads.

The task may therefore be considered to be "complete" before any processing in daughter threads has finished. When Ant terminates, any daughter threads are then killed without ceremony. The easiest way to fix this issue is to append the fork="true" attribute to the task in question thus:

 
<!-- Run the connected deadlock example -->
 
<target name="deadlock" depends="dist">
 
   <java classname="com.hibernatebook.chapter09.deadlock.GenerateDeadlock" classpathref="classpath.run" fork="true">
 
      <arg value="42"/>
 
   </java>
 
</target>
 

This forces Ant to start the task in a new JVM process. The normal thread termination rules will then be obeyed, and Ant will only proceed with the next build task (or exit) when the forked JVM process terminates.
